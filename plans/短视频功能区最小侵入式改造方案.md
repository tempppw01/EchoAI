# 短视频功能区最小侵入式改造方案

## 目标

在不破坏 EchoAI 现有工作区结构的前提下，把短视频功能区逐步接入进来，并且尽量减少：

- 大范围重构
- 无关模块改动
- 新旧逻辑耦合失控
- 现有功能区回归风险

## 一、总体策略

采用 **先复用、后扩展、最后抽象** 的路线，而不是一开始大改架构。

也就是：

1. 先挂到现有工作区体系内
2. 先借现有 `videoScript` 能力演进
3. 先把业务价值跑通
4. 再根据复杂度决定是否拆更多结构

## 二、推荐改造顺序

### 第一步：先不改顶层框架
保持以下内容不动或少动：

- `app/layout.tsx`
- `app/page.tsx`
- `Workspace` 的整体外壳结构
- 现有全局设置中心
- 现有会话系统主逻辑

这样能保证工作区整体稳定。

### 第二步：从现有 `videoScript` 模块位扩展
先不要直接新增一套完全独立的“短视频系统”，而是：

- 以 `videoScript` 为当前接入锚点
- 逐步把它升级成短视频功能区入口
- 先扩展其输入结构和结果展示结构

这是侵入最小的做法。

### 第三步：新增专属结果组件，而不是重写通用组件
应优先新增：

- 短视频脚本结果卡片
- 爆款拆解结果卡片
- 分镜建议卡片
- 风格分析结果卡片

而不是先去大改：

- `MessageList`
- `ChatList`
- 全局布局

### 第四步：在数据层做轻量扩展
可以在当前 `ChatSession` / `videoScriptPreset` 基础上，逐步增加：

- 短视频任务输入字段
- 结构化结果字段
- 模板引用字段

但不要在第一轮就把整个 store 重写成复杂领域模型。

## 三、哪些地方尽量不要先动

首期应尽量避免：

1. 重做整体路由体系
2. 重做全局状态管理架构
3. 重做消息系统基础逻辑
4. 引入新的重型编辑器框架
5. 批量改动所有现有功能区组件
6. 为短视频功能区单独开第二套设置中心

这些改动都太重，不符合 MVP 阶段要求。

## 四、最小侵入式实施建议

### 方案 A：模块内增强（首选）

做法：
- 保持 `Workspace` 不变
- 保持当前模块导航主结构不变
- 把 `videoScript` 模块升级为短视频功能区入口
- 在该模块内新增结构化输入面板与结果面板

优点：
- 风险最低
- 改动范围最小
- 最容易快速验证

### 方案 B：模块内子页面（第二阶段）

做法：
- 仍在 `Workspace` 内
- 在短视频模块主内容区内部做 tab / 子导航
- 承载脚本生成、爆款拆解、风格参考、剪辑思路等子页

优点：
- 比方案 A 更清晰
- 仍不需要重做顶层架构

### 当前建议
首期先做方案 A，后续再自然演进到方案 B。

## 五、回归风险控制建议

为了防止影响现有功能区，实施时应遵守：

1. 改动尽量局限在短视频模块相关文件
2. 新增能力优先以新组件形式落地
3. 公共组件只做必要增强，不做无关重构
4. 若要改 store，优先做新增字段，不先做大规模结构迁移
5. 每一步都保持现有 `chat`、`copywriting`、`roleplay`、`training`、`image` 可正常进入

## 六、总结

最小侵入式改造的核心不是“完全不改”，而是：

- 先改最靠近短视频模块的地方
- 先做新增，不做大改
- 先复用底座，不重造底座
- 先验证闭环，再决定抽象升级

这条路线最符合当前 EchoAI 的代码现状，也最适合一口气把规划推进完后进入实现阶段。
