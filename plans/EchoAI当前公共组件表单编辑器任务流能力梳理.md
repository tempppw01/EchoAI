# EchoAI 当前公共组件、表单能力、编辑器能力与任务流能力梳理

## 一、公共组件现状

当前 EchoAI 已经具备一批可复用的基础组件，虽然数量不算多，但足以支撑工作区继续扩展。

### 1. 基础 UI 组件
当前明确存在的基础组件包括：

- `components/ui/button.tsx`
- `components/ui/input.tsx`
- `components/ui/textarea.tsx`

这些组件已经承担了：

- 统一按钮样式
- 统一输入框样式
- 统一文本域样式
- 通过 `className` 做轻量扩展

### 2. 工作区级复用组件
当前已经存在一批可在多个功能区复用的工作区组件：

- `ChatList`：会话列表与搜索
- `MessageList`：消息流展示与导出
- `ChatComposer`：统一输入与发送面板
- `SettingsCenter`：统一设置中心
- `SettingsTabs`：设置分栏

这些组件说明当前项目已经具备“工作区复用件”基础，而不是所有逻辑都写死在页面中。

## 二、表单能力现状

当前表单能力主要集中在设置中心和视频脚本预设输入两处。

### 1. 表单基础栈
当前已使用：

- `react-hook-form`
- `zod`
- `@hookform/resolvers/zod`

对应文件：
- `components/settings/settings-center.tsx`
- `components/settings/settings-form-schema.ts`

### 2. 已具备的表单能力
当前可以确认已有：

- 表单状态管理
- Zod 校验
- 默认值回填
- 实时持久化部分字段
- 提交保存
- 字段级输入控件
- 基础 select / input / number 输入组合

### 3. 对短视频功能区的意义
这意味着短视频功能区后续要做：

- 主题输入
- 平台选择
- 时长选择
- 内容类型选择
- 风格参考输入
- 模板参数输入

并不需要重新搭建一套表单基础设施，可以直接延续当前表单栈。

## 三、编辑器能力现状

这里的“编辑器能力”目前还不算真正富文本编辑器，更准确说是“聊天输入 + 文本编辑”能力。

### 1. 当前已有能力
在 `ChatComposer` 和 `MessageList` 中，已经具备：

- 多行文本输入
- 用户消息编辑
- 生成结果 Markdown 渲染
- 附件占位与图片 data URL 预览
- 视频脚本预设输入区
- 导出文本结果

### 2. 当前编辑能力的边界
当前还没有：

- 真正的富文本编辑器
- 所见即所得块编辑器
- 结构化段落级编辑器
- 结果模块拖拽重排能力

### 3. 对短视频功能区的启发
短视频功能区首期不需要上重型编辑器，现有文本输入 + 结构化结果展示已经足够支撑前几轮验证。

后续如果要做“局部重写”“模块级优化”，更适合在现有文本编辑基础上做轻量增强，而不是直接引入复杂编辑器框架。

## 四、任务流能力现状

当前项目虽然没有独立命名成 workflow 引擎，但已经存在基础任务流雏形。

### 1. 会话任务流
在 `stores/chat-store.ts` 中，已经实现了：

- 创建会话
- 选择会话
- 发送消息
- 停止生成
- 重试消息
- 重新生成回复
- 删除消息
- 编辑消息
- 会话置顶 / 重命名 / 删除

这本质上就是当前工作区的通用任务流基础。

### 2. 模式化任务流
当前不同 mode 已有不同处理逻辑：

- `chat`：通用对话
- `copywriting`：文案生成
- `videoScript`：视频脚本生成
- `roleplay`：角色扮演
- `training`：学习训练闭环
- `image/proImage`：绘图相关

说明当前系统已经支持“按模式切换任务行为”。

### 3. 训练模式的专用任务流
训练模式尤其说明：

- 当前系统已具备“模式特化工作流”能力
- 可以做多轮状态推进
- 可以做专属评分、问题、反馈循环

这对短视频功能区很关键，因为后续脚本生成、爆款拆解、风格参考、分镜建议，其实也可以设计成分阶段工作流。

## 五、当前能力的总体判断

### 已经具备的基础
- 基础 UI 组件
- 工作区级复用组件
- 表单栈
- Markdown 展示
- 文本输入与轻编辑能力
- 会话型任务流
- 按 mode 区分行为的能力

### 仍然偏弱的部分
- 结构化结果组件还不够丰富
- 真正的业务表单组件较少
- 专门面向模板、结果卡片、分镜表的组件还没有建立
- 复杂工作流目前更多藏在 store 逻辑里，尚未形成清晰的业务层抽象

## 六、对短视频功能区接入的直接结论

因此，短视频功能区接入时可以采用以下策略：

1. 复用现有 UI 基础组件和工作区容器
2. 复用当前表单栈来搭建脚本/风格/拆解输入表单
3. 复用现有会话与消息任务流作为基础生成流程
4. 在此基础上新增短视频专属结果组件，而不是重做底层

## 七、总结

当前 EchoAI 的前端能力还不算“组件体系非常成熟”，但已经具备：

- 可复用基础组件
- 可用的表单能力
- 轻量编辑能力
- 可扩展的模式化任务流

这已经足够支撑短视频功能区进入下一阶段的接入方案设计与页面信息架构设计。
