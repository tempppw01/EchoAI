# 短视频功能区复用边界说明

## 目标

明确短视频功能区在接入 EchoAI 时，哪些能力应该直接复用，哪些能力应做功能区内扩展，哪些能力不应在首期重复建设。

## 一、必须复用的基础能力

以下能力应直接复用 EchoAI 现有基础设施：

### 1. 工作区布局与导航
- 复用 `components/layout/workspace.tsx` 提供的统一工作区框架
- 复用顶部栏、左侧模块区、主内容区、设置入口、移动端侧栏机制
- 不单独新建第二套产品级外壳

### 2. 会话与历史记录机制
- 复用 `stores/chat-store.ts` 中已有的 session 管理能力
- 复用新建会话、切换会话、删除、重命名、置顶、时间排序等机制
- 短视频功能区的生成结果，优先作为一种新会话类型或现有 `videoScript` 会话扩展承载

### 3. 模型与接口配置
- 复用 `stores/settings-store.ts` 中的模型、Base URL、API Key、参数配置
- 复用现有 OpenAI Compatible 请求链路
- 不为短视频功能区单独再做一套 provider 配置页

### 4. 通用 UI 状态
- 复用 `stores/ui-store.ts` 中的侧栏开关、设置弹层等 UI 状态管理
- 复用现有按钮、输入框、文本域、卡片、弹窗等基础 UI 组件
- 保持加载态、错误态、空状态与全局风格一致

## 二、应在短视频功能区内独立扩展的能力

以下能力不适合完全复用现有通用聊天形态，需要在短视频功能区内新增专属结构：

### 1. 结构化结果展示
- 脚本生成结果不能只是一段普通聊天文本
- 需要新增结构化结果区，例如：标题、钩子、正文、CTA、分镜、字幕建议
- 这部分属于短视频功能区专属展示层

### 2. 短视频专用输入表单
- 通用聊天输入框不够表达平台、时长、内容类型、目标受众、风格参考等字段
- 需要在短视频功能区中增加专用输入表单或预设面板
- 但底层发送与生成机制可继续复用现有请求流

### 3. 模板与风格参考能力
- 短视频模板、平台策略、风格参考、爆款拆解结果等，应作为短视频功能区的专属业务对象
- 这些对象可接入现有状态体系，但不应强塞进普通聊天字段中

### 4. 导出与执行结果格式
- 短视频功能区后续需要导出脚本、分镜表、剪辑建议
- 这属于业务层扩展，不应污染普通聊天结果结构

## 三、首期不应重复建设的能力

为了保持最小侵入，首期不应重复建设以下内容：

- 不单独建设新的登录/鉴权体系
- 不单独建设新的设置中心
- 不单独建设新的模型管理页
- 不单独建设新的历史记录系统
- 不单独建设新的全局布局系统
- 不单独建设重型视频编辑器、时间线剪辑器、素材轨道系统

## 四、建议的边界划分方式

建议按“三层复用”来实施：

1. **底座层复用**：布局、设置、请求层、状态管理、基础组件
2. **业务层扩展**：短视频专属表单、模板、风格分析、爆款拆解、分镜结果
3. **表现层定制**：结构化结果卡片、模块化输出面板、导出能力

## 五、首期落地结论

首期最稳妥的做法是：

- 复用 EchoAI 的工作区底座
- 在 `videoScript` 现有基础上扩展为更完整的短视频功能区
- 新增短视频专属输入与结果结构
- 避免把短视频能力做成第二套独立系统

这样可以在最小改动下完成短视频功能区接入，同时为后续 0.4.0 ~ 0.14.0 的具体功能开发留出清晰边界。
