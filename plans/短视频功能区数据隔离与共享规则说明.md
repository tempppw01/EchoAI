# 短视频功能区数据隔离与共享规则说明

## 目标

明确短视频功能区与 EchoAI 其他功能区之间，哪些数据应该共享，哪些数据必须隔离，避免后续出现状态污染、结果串用和边界混乱。

## 一、总体原则

采用 **共享底座、业务隔离、必要互通** 的规则：

- 底层配置可以共享
- 业务结果需要按功能区隔离
- 只有明确对用户有帮助的数据，才允许跨功能区复用

## 二、必须共享的数据

### 1. 全局设置数据
以下数据应在所有功能区共享：

- API Key
- Base URL
- 模型列表
- 默认模型
- 温度、最大 Token、流式输出等参数
- 主题设置与基础 UI 偏好

原因：这些属于 EchoAI 工作区级配置，不应因功能区不同而重复维护。

### 2. 全局 UI 框架状态
以下状态应共享：

- 工作区侧栏开关状态
- 设置中心开关状态
- 主题切换状态
- 通用导航结构

原因：这些属于统一工作区体验的一部分。

### 3. 可复用的参考输入
在用户明确操作时，以下内容可以跨功能区复用：

- 文案生成功能区中的可复用内容片段
- 通用对话中沉淀的选题、卖点、用户画像信息
- 短视频功能区中提炼出的风格模板
- 爆款拆解形成的结构模板

但前提是：
- 必须经过用户显式选择或明确触发
- 不能默认自动把其它功能区的内容灌入当前任务

## 三、必须隔离的数据

### 1. 会话数据必须按功能区隔离
短视频功能区的会话记录不应与以下模块混在一起：

- 通用对话
- 角色扮演
- 学习型聊天
- 专业绘图

原因：这些功能区的上下文目标完全不同，若混用会导致历史记录混乱、上下文污染、生成结果失真。

建议规则：
- 继续使用 `ChatSession.mode` 做功能区隔离
- 短视频相关会话只归属于 `videoScript` 或后续独立短视频 mode
- 会话列表可以统一展示，但内容上下文必须按 mode 独立读取

### 2. 功能区专属业务字段必须隔离
以下数据应视为短视频功能区专属字段：

- 平台偏好
- 时长偏好
- 内容类型
- 风格模板引用
- 爆款拆解结果
- 分镜结果
- 剪辑建议结果

这些字段不能直接塞进其他功能区的通用字段里，否则会造成数据结构混乱。

### 3. 生成结果必须隔离存储
以下结果应以短视频专属结果对象或扩展字段形式保存：

- 结构化脚本
- 爆款拆解结构
- 分镜与剪辑建议
- 风格分析结果

不能只把这些内容当作一段普通 assistant 文本长期混存，否则后续无法支持结构化展示、导出与复用。

## 四、允许共享但需要显式控制的数据

以下数据允许跨模块共享，但需要显式入口：

### 1. 从文案模块导入到短视频模块
场景：用户在文案模块先写出一版卖点文案，再转到短视频模块生成脚本。  
规则：必须通过“导入为脚本素材”或类似明确操作触发。

### 2. 从短视频模块导出到其他模块
场景：将短视频脚本中的标题、口播稿导出到文案模块继续润色。  
规则：必须通过“复制到文案任务”或明确导出动作触发。

### 3. 风格模板跨任务复用
场景：某条短视频任务提炼出风格模板，后续任务继续调用。  
规则：可以共享，但应作为模板资产显式保存，而不是偷偷继承上一条任务上下文。

## 五、首期最小落地规则

首期建议先落这几条最核心规则：

1. 共享全局设置与工作区框架
2. 按 `mode` 隔离不同功能区会话
3. 短视频专属字段单独扩展，不污染通用聊天字段
4. 跨模块复用必须通过用户显式操作触发
5. 短视频结果逐步走向结构化存储，而不是只保留纯文本

## 六、后续实施建议

随着短视频功能区继续发展，建议后续进一步拆分：

- `session` 作为通用会话容器
- `shortVideoDraft` 作为短视频任务对象
- `styleTemplate` 作为风格模板对象
- `viralPatternTemplate` 作为爆款结构模板对象

这样既保留 EchoAI 的统一会话体验，又能保证短视频业务数据结构足够清晰。

## 结论

短视频功能区与其他功能区的关系，不是完全隔离，也不是完全打通，而是：

- 配置层共享
- 会话层隔离
- 模板层可控复用
- 结果层结构化沉淀

这套规则可以保证首期接入足够轻，同时不给后续功能扩展埋雷。
