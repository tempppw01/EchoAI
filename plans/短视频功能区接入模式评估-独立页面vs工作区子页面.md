# 短视频功能区接入模式评估：独立页面 vs 工作区子页面

## 结论

基于当前 EchoAI 结构，短视频功能区更适合采用 **工作区子页面 / 工作区内模块扩展模式** 接入，而不是走独立页面模式。

## 一、为什么不优先选独立页面

如果短视频功能区直接做成独立页面，会出现几个问题：

1. **与当前产品结构不一致**
   - 当前 EchoAI 明显是统一工作区结构
   - 现有能力都围绕 `Workspace` 这个统一容器展开
   - 若短视频突然独立成另一套页面，会破坏当前产品组织方式

2. **重复建设成本高**
   - 需要额外维护页面入口、导航关系、状态同步、设置入口、历史记录衔接
   - 当前还没有足够强的页面级业务架构来支撑独立子系统

3. **不利于复用现有能力**
   - 当前会话、设置、模块切换、UI 框架都集中在工作区内
   - 独立页面会让这些复用链条变长，增加维护成本

4. **与首期目标不匹配**
   - 短视频功能区首期是 MVP 验证，不是大规模产品拆分
   - 在这个阶段单独立页面，投入过重

## 二、为什么更适合工作区子页面模式

### 1. 符合当前工作区组织方式
当前已经有：

- 通用对话
- 文案生成
- 视频脚本
- 角色扮演
- 学习型聊天
- 专业绘图

这本质就是工作区内的模块体系。  
短视频功能区应该沿着这条线继续扩展，而不是跳出去重起一套入口。

### 2. 复用链路最短
若采用工作区子页面模式，可以直接复用：

- `Workspace` 外壳
- 左侧模块导航
- 设置中心
- 会话历史
- 模式切换逻辑
- 现有 store 结构

### 3. 便于渐进式演化
短视频功能区可以分阶段演进：

- 第一阶段：先在 `videoScript` 模块位上增强
- 第二阶段：扩展成短视频一级模块
- 第三阶段：在模块内再拆脚本、拆解、风格、分镜等子页面

这条路径比直接独立页面更稳。

## 三、推荐的接入方式

推荐采用：

### 方案：工作区一级模块 + 模块内子页面

层级建议：

- EchoAI
  - 工作区
    - 短视频功能区
      - 脚本生成
      - 爆款拆解
      - 风格参考
      - 剪辑思路
      - 历史记录

### 首期最小落地方式

首期可以更保守：

- 继续复用现有 `videoScript` 模块位
- 先在该模块下做信息架构升级
- 不急着马上改站点级路由
- 等到功能和结构稳定后，再决定是否单独暴露 `/short-video` 之类的路径

## 四、什么时候才考虑独立页面

只有在以下情况同时出现时，才建议进一步评估独立页面：

- 短视频功能区内部页面明显增多
- 拥有自己的完整业务对象体系
- 拥有自己的结果页、模板页、历史页、项目页
- 工作流复杂度明显超过当前通用工作区模块
- 需要更强的 URL 可分享性或更独立的页面状态管理

换句话说：

**独立页面不是现在不能做，而是现在没必要先做。**

## 五、当前阶段的最终建议

当前阶段推荐：

1. 采用工作区子页面 / 模块扩展接入
2. 保持短视频功能区挂在 EchoAI 工作区内部
3. 先做模块内信息架构，不急着重做顶层路由
4. 先验证业务闭环，再考虑页面拆分

## 六、总结

在当前 EchoAI 的真实代码结构下，最合理的路线不是：

- 直接独立页面
- 直接拆成独立产品

而是：

- 先作为工作区内模块增强接入
- 再在模块内部逐步页面化
- 最后根据复杂度决定是否独立路由化

这条路线最省成本，也最符合当前项目现状。
